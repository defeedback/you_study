# AI Agent 岗位能力地图

本文件把岗位 JD 中的技术栈要求转换成可学习、可练习、可做项目的能力模块。后续制定个人学习路线时，应结合用户画像再决定每个模块的深度和时间安排。

## 能力模块总览

| 能力模块 | 对应岗位需求 | 优先级 | 推荐产出 |
|---|---|---|---|
| 编程与后端工程基础 | Python、API、数据库、缓存、Linux、Docker | P0 | 可运行的 Web API 服务 |
| LLM API 与 Prompt Engineering | 模型 API、Prompt、CoT、Few-shot、ReAct、输出质量优化 | P0 | Prompt 模板库和模型调用 Demo |
| Function Calling / Tool Calling | 工具调用、API 集成、权限控制、异常处理 | P0 | 可调用外部工具的 Agent |
| RAG 知识库系统 | 数据清洗、分块、Embedding、检索、Rerank、幻觉控制 | P0 | 知识库问答系统 |
| Agent 核心架构 | 任务规划、记忆管理、工作流、多轮对话、多智能体 | P0 | 多步骤任务执行 Agent |
| Agent 框架与编排 | LangChain、LangGraph、LlamaIndex、Dify、AutoGen、CrewAI | P0/P1 | 框架版 Agent 项目 |
| 工程化与部署 | FastAPI、MySQL、Redis、Docker、日志、监控、流式响应 | P1 | 可部署的 Agent 后端服务 |
| LLMOps 与评测 | 准确率、召回率、任务成功率、Token 成本、日志追踪 | P1 | Agent 评测与监控报告 |
| 业务落地与产品思维 | 需求拆解、流程设计、指标定义、跨团队协作 | P0 | 业务 Agent 方案文档 |
| 简历项目与面试表达 | 项目包装、架构图、技术难点、结果复盘 | P1 | 简历项目描述和面试问答材料 |

## 1. 编程与后端工程基础

### 对应 JD 要求

- Python 是最高频语言，Java / Go / TypeScript / Node.js 是常见补充。
- 多个岗位要求 FastAPI / Flask / Django / Spring Boot、MySQL、Redis、Linux、Docker。
- 后端岗位强调 API 设计、接口联调、系统稳定性、缓存、消息队列和可维护性。

### 应掌握的核心知识

- Python 基础：语法、文件 IO、异常处理、模块化、虚拟环境
- HTTP / RESTful API / JSON
- FastAPI 或同类 Web 框架
- MySQL / PostgreSQL 基础 SQL
- Redis 缓存基础
- Linux 常用命令
- Docker 与 docker-compose 基础

### 最小实践任务

做一个 `LLM Chat API`：

- 提供 `/chat` 接口
- 接收用户问题
- 调用一个 LLM API
- 返回普通响应和流式响应
- 记录请求、响应、错误日志

### 可升级项目

- 加入用户会话表和历史消息存储。
- 加入 Redis 缓存和限流。
- 使用 Docker 部署。
- 输出接口文档和架构图。

### 常见难点

- 只会写脚本，不会封装成服务。
- 不理解同步 / 异步、流式响应和长任务处理。
- 忽视错误处理、日志和配置管理。

## 2. LLM API 与 Prompt Engineering

### 对应 JD 要求

- 大部分岗位要求了解 OpenAI、Claude、DeepSeek、Qwen、GLM 等模型 API。
- Prompt Engineering、指令工程、CoT、Few-shot、ReAct 在多份 JD 中反复出现。
- 产品和运营类岗位也要求能设计 Prompt 策略与工作流。

### 应掌握的核心知识

- Chat Completions / Messages API 的基本结构
- System / User / Assistant 角色
- Prompt 模板与变量注入
- 输出格式约束
- Few-shot 示例
- CoT / ReAct 的使用边界
- Prompt 评估与版本管理

### 最小实践任务

为同一个任务设计 3 个 Prompt 版本：

1. 简单问答版
2. 结构化输出版
3. 带 Few-shot 示例版

记录每个版本的输出质量、稳定性和问题。

### 可升级项目

- 做一个 Prompt 模板管理小工具。
- 加入自动化评估样例集。
- 对比不同模型在同一 Prompt 下的表现。

### 常见难点

- 把 Prompt 当成一次性文本，不做版本管理。
- 没有清晰定义输入、输出、约束和评价标准。
- 过度依赖复杂 Prompt，忽视数据、工具和流程设计。

## 3. Function Calling / Tool Calling

### 对应 JD 要求

- 多个 Agent 岗位明确要求 Function Calling、Tool Calling、工具调用、业务 API 集成。
- Agent 能否落地，很大程度取决于能否安全、稳定地调用外部工具。

### 应掌握的核心知识

- 工具 schema 设计
- 参数校验
- 工具调用结果回填给模型
- 工具错误处理
- 权限和安全边界
- 多工具选择与调用链

### 最小实践任务

做一个工具调用 Agent，至少包含 3 个工具：

- 时间 / 日期工具
- 本地文件检索工具
- 简单计算或数据查询工具

### 可升级项目

- 接入真实业务 API。
- 加入工具调用日志。
- 支持多步工具调用。
- 对工具调用失败进行重试或降级。

### 常见难点

- 工具描述不清晰，模型不知道何时调用。
- 参数 schema 不稳定，导致调用失败。
- 没有控制工具权限，容易把 Agent 做成不安全系统。

## 4. RAG 知识库系统

### 对应 JD 要求

- RAG、知识库、向量数据库、Embedding、Rerank 是最常见关键词之一。
- 办公、客服、教育、广告审核、新零售、企业知识库等场景都依赖 RAG。

### 应掌握的核心知识

- 文档解析
- 数据清洗
- Chunking 策略
- Embedding 模型
- 向量数据库
- 相似度检索
- 混合检索
- Rerank
- 引用来源
- 召回率、准确率和幻觉控制

### 最小实践任务

做一个本地知识库问答系统：

- 支持上传 Markdown / PDF / TXT 文档
- 完成分块和向量化
- 支持相似度检索
- 调用 LLM 生成答案
- 返回引用片段

### 可升级项目

- 对比不同 chunk size 的效果。
- 加入 Rerank。
- 加入检索评估集。
- 支持多知识库和权限隔离。

### 常见难点

- 只会“向量库 + LLM”，不理解数据清洗和分块。
- 不评估召回效果，只看最终回答。
- 无法解释为什么答案错、漏、幻觉。

## 5. Agent 核心架构

### 对应 JD 要求

- 岗位反复提到任务规划、任务分解、记忆管理、工具调用、多轮对话、工作流编排、多智能体协作。
- 这是 AI Agent 开发区别于普通 LLM 应用开发的核心。

### 应掌握的核心知识

- Agent = 模型 + 状态 + 工具 + 规划 + 反馈
- 任务规划与步骤拆解
- 短期记忆与长期记忆
- 工具选择和执行
- 观察结果与反思修正
- 多轮上下文管理
- 多 Agent 角色协作

### 最小实践任务

实现一个简单 Agent 循环：

1. 接收用户目标
2. 拆解任务
3. 选择工具
4. 执行工具
5. 观察结果
6. 输出最终答案

### 可升级项目

- 加入长期记忆。
- 加入任务状态机。
- 加入失败重试。
- 拆成规划 Agent、执行 Agent、评审 Agent。

### 常见难点

- Agent 流程不可控，容易无限循环。
- 没有状态管理，无法恢复任务。
- 工具调用和自然语言推理边界不清。

## 6. Agent 框架与编排

### 对应 JD 要求

- 高频框架包括 LangChain、LangGraph、LlamaIndex、AutoGen、Dify、CrewAI、OpenClaw、Spring AI、Semantic Kernel、AgentScope。
- JD 对框架的要求通常不是“背 API”，而是能用框架完成业务落地。

### 应掌握的核心知识

- LangChain：Prompt、Tool、Retriever、Chain / Runnable
- LangGraph：状态机、多节点工作流、可控 Agent
- LlamaIndex：数据连接、索引、RAG
- Dify / Coze：低代码 Agent 和工作流原型
- AutoGen / CrewAI：多 Agent 协作思想

### 最小实践任务

用 LangChain 或 LangGraph 重写前面手写的 Agent：

- 明确状态
- 明确工具节点
- 明确结束条件
- 输出执行轨迹

### 可升级项目

- 同一个业务场景分别用 LangChain、LangGraph、Dify 实现，对比优缺点。
- 将 RAG、工具调用和工作流组合成完整 Agent。

### 常见难点

- 只会调用框架示例，不理解底层 Agent 流程。
- 为了用框架而用框架，项目结构反而复杂。
- 不会排查框架内部状态和工具调用错误。

## 7. 工程化与部署

### 对应 JD 要求

- 多个岗位要求测试、部署、日志、监控、错误恢复、高并发、高可用。
- 企业不会只看 Notebook Demo，更关注系统能否稳定运行。

### 应掌握的核心知识

- 配置管理
- 日志记录
- 异常处理
- 流式响应
- 异步任务
- Docker 部署
- 基础监控
- API 文档
- 简单 CI/CD 概念

### 最小实践任务

将 RAG / Agent 项目改造成可运行服务：

- FastAPI 接口
- `.env` 配置
- 日志文件
- Dockerfile
- README 使用说明

### 可升级项目

- 增加 Redis 缓存。
- 增加任务队列。
- 增加调用链日志。
- 增加服务健康检查。

### 常见难点

- Demo 能跑，但换机器就跑不起来。
- 没有日志，出错无法定位。
- 没有成本和延迟意识。

## 8. LLMOps 与评测

### 对应 JD 要求

- JD-04、JD-26、JD-27、JD-30 等强调指标监控、模型评测、Token 成本、日志、异常处理。
- 这是从个人 Demo 走向企业项目的关键能力。

### 应掌握的核心知识

- 测试集 / 评估集构建
- 任务成功率
- RAG 召回率与答案准确性
- Prompt 版本对比
- Token 成本统计
- 延迟监控
- 日志追踪

### 最小实践任务

为一个 RAG 或 Agent 项目设计评测表：

- 10 条典型问题
- 标准答案或评价维度
- 模型回答
- 是否命中知识来源
- 是否完成工具调用
- Token 成本

### 可升级项目

- 自动化运行评测集。
- 对比不同模型和 Prompt 版本。
- 输出评测报告。

### 常见难点

- 只凭主观感觉判断效果。
- 不区分模型错误、检索错误、Prompt 错误和工具错误。
- 忽视成本、延迟和稳定性。

## 9. 业务落地与产品思维

### 对应 JD 要求

- 多数 JD 都提到跨团队协作、业务场景落地、需求拆解、指标监控和技术方案设计。
- AI Agent 项目必须解决具体业务问题，否则很难形成有竞争力的简历项目。

### 应掌握的核心知识

- 用户画像
- 业务流程拆解
- Agent 角色定义
- 输入输出定义
- 工具和数据源设计
- 验收指标
- 项目复盘

### 最小实践任务

选择一个业务场景，写一份 Agent 方案：

- 业务痛点
- 用户角色
- Agent 能力边界
- 数据源
- 工具列表
- 流程图
- 验收指标

### 可升级项目

- 把方案做成可运行 Demo。
- 记录真实测试用例和改进过程。
- 整理成简历项目和面试材料。

### 常见难点

- 技术很多，但业务问题不清楚。
- 项目没有验收标准，无法说明效果。
- 面试时只会罗列技术栈，不会讲业务价值。

## 10. 简历项目与面试表达

### 对应 JD 要求

- 多个岗位明确偏好“完整项目经验”“从 0 到 1 落地”“开源项目”“个人 Demo”。
- 学习过程必须沉淀为可展示的项目，而不只是笔记。

### 应掌握的核心知识

- 项目 README 写作
- 架构图表达
- 技术选型说明
- 难点与解决方案
- 效果评估
- 面试 STAR 表达

### 最小实践任务

为每个阶段项目输出：

- 项目背景
- 技术栈
- 架构图
- 核心模块
- 运行方式
- 效果截图或测试结果
- 遇到的问题和解决方案

### 可升级项目

- 将 2-3 个项目包装成简历项目。
- 为每个项目准备 10 个面试追问与回答。
- 总结项目可扩展方向。

### 常见难点

- 项目做了但没有文档和截图。
- 不会把学习项目转成岗位语言。
- 面试时说不清系统边界、技术难点和取舍。

## 建议的项目递进路线

1. LLM API 聊天服务
2. Prompt 模板与评估小工具
3. Function Calling 工具调用 Agent
4. RAG 知识库问答系统
5. LangGraph 多步骤 Agent
6. 企业场景 Agent：办公 / 客服 / 教育 / 新零售任选一个
7. 可部署 Agent 后端服务：日志、评测、Docker、README
8. 简历项目包装与面试问答材料
