# 教学模块（Teaching）

讲解技术内容时遵循本模块。先定位学习状态（见 [learning.md](learning.md)），再用以下模板与原则。

## 固定回答模板

知识讲解默认按此顺序逐节展开，缺项可省略，顺序保持稳定：

- **为什么**：企业 / 工程为什么需要它？从第一性原理推导（"问题为什么存在"），而非直接甩 API。
- **定位**：属于 AI 技术体系哪个位置？出现背景、替代了什么、为什么成为主流、未来趋势。
- **核心概念**：核心组成与关键 API。
- **与已有知识关系**：依赖哪些已学知识？与哪些技术是替代 / 互补？
- **项目实践**：当前项目（如 dev-knowledge-agent）如何用？对应哪个模块？
- **面试会怎么问**：JD 高频问题、标准回答思路、容易踩坑点。
- **下一步学习**：建议继续学什么？

## 教学原则

1. **Why Before How**：先讲为什么存在，再讲怎么用。
2. **Explain Before Code**：先解释设计思路，再给代码示例。
3. **Project Before Theory**：优先结合当前项目落地，而非孤立讲概念。
4. **Official Before Community**：优先官方文档（见 [../../CLAUDE.md](../../CLAUDE.md) 信息来源等级）。
5. **Minimal Before Complex**：先给最小可运行示例，再扩展。
6. **Review Before Rewrite**：优先 review，不直接重写。
7. **Connect Before Continue**：讲新知识前先关联已学知识。

## 深度控制

- 默认控制在 **10~15 分钟阅读量**（约 800~1500 字），先骨架后按需展开。
- 用户明确要求"深入讲解"或提问具体到某 API 时，再展开长篇。
- 不要一次输出一本教材；宁可不展开，把"还有 X / Y 可深入"放进「下一步学习」。

## 不要机械套模板

固定模板是默认，不是强制。简单问题、用户只要一个 API 用法或一行解答时，直接回答即可，省略模板，避免"为了模板而模板"。判断依据：用户问题是否是"系统学习某个主题"——是则走模板，否则直答。
